# trip-road テレメトリ分析プロンプト集

`fetch_entries.sh` で取得した JSONL を Claude（claude.ai または API）に貼り付けて、
以下のプロンプトと組み合わせて使う。Athena でなく LLM 中心で分析する方針の意図は
`docs/knowledge.md` 4.7 セクション参照。

---

## 重要な前提（毎回 LLM に伝える）

```
trip-road は GPS で現在の市町村を判定し、Claude Haiku が「土地のたより」
と呼ばれる短い解説（120〜180字）を生成する個人用旅アプリです。
1 entry = 1 つの解説が表示された記録です。

dwell_ms（画面に表示されていた時間）は、ユーザが集中して読んだ時間ではなく、
単に画面が開きっぱなしだった時間です。電車内で景色を見ていたり、他の作業に
切り替えたりするので、dwell_ms から「面白さ」を判断しないでください。
あくまで参考情報として扱い、結論は description の本文の品質を主眼に出してください。
```

---

## 1. 汎用表現の検出

> 添付した JSONL は trip-road が生成した「土地のたより」のテレメトリです。
> 各 entry の description は Claude Haiku が生成した本文です。
>
> **問題意識**: 「春野菜が美味しいです」「四季の移ろいが楽しめます」のように、
> どの市町村にも当てはまる汎用フレーズが頻出している懸念があります。
>
> 以下を抽出してください：
> 1. 複数の muni_code に出現する汎用フレーズを頻度順に TOP 15
> 2. 各フレーズが、地理的に離れた市町村でどれだけ使い回されているか
> 3. 特に汎用度の高い entry を 5 個ピックアップし、その muni_code・description 全文・
>    具体性を高めるとしたら何を入れるべきかの提案
>
> ※ 季節（season）は同じでも市町村が異なるなら、季節依存表現は OK です。
>   問題は「市町村が変わっても通用してしまう」表現です。

---

## 2. system prompt 改善案

> 添付の JSONL を見て、`workers/src/anthropic.js` の system prompt をどう改善すべきか、
> 具体提案を 3 つ出してください。
>
> 各提案には以下を含めてください：
> - 改善ポイント（例: 汎用表現の禁止、固有名詞の必須化）
> - prompt に追加すべき具体テキスト案（です・ます調、観光ガイド口調を維持）
> - その変更で改善が期待される entry の具体例（trace_id 引用）
> - 副作用として悪化しうる点（例: 出力が硬くなる、文字数オーバー）

---

## 3. 再生成すべきキャッシュ候補

> trip-road は description を `{muni_code}_{season}` でキャッシュしているため、
> 一度品質の低い出力が生成されると、その季節の間ずっとそれが使われ続けます。
>
> 添付 JSONL から、再生成して品質を上げるべき entry を TOP 10 で挙げてください：
> - muni_code / season / description（全文）/ 何が問題か（汎用すぎる、誤情報の疑い、薄い、など）
> - 優先度（高/中/低）
>
> 結果は手動でキャッシュをクリア → 次回訪問時に新しい出力を得る運用に使います。

---

## 使い方の例

1. `bash docs/analysis/fetch_entries.sh` を実行
2. 出力された `docs/analysis/data/entries-YYYYMMDD-HHMMSS.jsonl` を開く
3. claude.ai に「重要な前提」+ プロンプト 1 を貼り、続けて JSONL 全体をペースト
4. 結果を読み、気になった部分を深掘りするためにプロンプト 2、3 を続ける
5. プロンプト 2 で得た system prompt 改善案は、`workers/src/anthropic.js` を直接編集 →
   `wrangler deploy` で反映 → しばらく実走 → 1 週間後に再度 fetch して比較

データが少ないうちは「サンプルが足りないので結論保留」が valid な答え。
無理に洞察を引き出さず、記録としての価値だけ取りに行く。
