# GPS ローカル旅ガイド 要件定義書

**バージョン**: 0.3  
**作成日**: 2026-04-22  
**ステータス**: 確定

---

## 1. プロジェクト概要

### 1.1 目的

電車・徒歩による旅行中に、現在地の市町村をリアルタイムで自動判定し、その土地の季節感ある解説をLLMで生成して表示するスマホWebアプリを構築する。GLG-01（GPSローカル旅ガイド端末）をスマホWebアプリとして再解釈したPoCプロジェクトである。

### 1.2 想定ユーザー

主なユーザーは電車旅行・徒歩移動が中心の個人ユーザーを想定する。新幹線・在来線・徒歩での移動中に画面を見て、通過する土地の解説を楽しむ「旅のお供」としての利用シーンを主に想定する。

### 1.3 スコープ（フェーズ1 / PoC）

本ドキュメントはフェーズ1（PoC）の要件のみを対象とする。フェーズ2以降の機能（道の駅表示・GPXエクスポート・オフライン対応）は別途定義する。

---

## 2. システム構成

### 2.1 全体アーキテクチャ

本アプリはフロントエンド（Cloudflare Pages）と薄いAPIプロキシ（Cloudflare Workers）の2層構成とする。バックエンドサーバーは持たない。外部への通信は以下の4種類である。

1つ目は**Cloudflare Workers（自作APIプロキシ）**で、ブラウザからのリクエストを受け取り、パスワード検証を行った上でAnthropicAPIキーを付与してClaude Haikuに転送する。APIキーはCloudflare Workersの環境変数として管理し、ブラウザには一切露出しない。2つ目は**Anthropic API（Claude Haiku）**で、Cloudflare Workers経由で呼び出す。3つ目は**国土地理院 逆ジオコーディングAPI**で、市町村判定のフォールバックとして低頻度で利用する。4つ目は**地理院タイル**で、地図表示のためにタイル画像を都度取得する。

なおAnthropicAPIはClaude Maxプランとは別サービスであり、Anthropicコンソール（https://console.anthropic.com）で別途アカウントを作成しAPIキーを発行する。利用料金は従量課金制で、利用前に最低5ドル分のクレジットを前払いする必要がある。

### 2.2 技術スタック

プロトタイプフェーズでは環境構築コストを最小化するため、**Vanilla JS + HTML + CSS**の構成を採用する。地図表示には**Leaflet.js 1.9.4**（BSD-2-Clause）、地図タイルには**地理院タイル（淡色地図）**を使用する。

市町村境界データには**国土数値情報 行政区域データ N03**（PDL1.0、商用利用可）を使用する。全国データをそのまま読み込むのではなく、後述の「市町村単位分割＋隣接市町村プリフェッチ」方式を採用する。

## 3. 機能要件

### 3.1 GPS取得と現在地表示

アプリ起動時にブラウザのGeolocation APIを使ってGPS取得を開始し、ユーザーに位置情報の使用許可を求める。許可が得られた後は `watchPosition` で継続的に位置情報を監視し、現在地を地図上にリアルタイムで表示する。速度情報はGeolocation APIの `coords.speed`（単位：m/s）を取得し、km/hに変換して画面に表示する。なお `coords.speed` が `null` を返す環境（一部のブラウザ）では「--」と表示する。

### 3.2 市町村の自動判定

市町村判定は**クライアント内でのPoint-in-Polygon判定を主軸**とし、国土地理院APIをフォールバックとして組み合わせる2段構えの設計とする。判定ロジックには**Turf.js の `booleanPointInPolygon`**を使用する。

データの取得方式は**市町村単位での動的ロード**とする。国土数値情報N03を市町村ごとに個別のGeoJSONファイルに分割し、Cloudflare Pagesに静的ファイルとして配置する。ファイル名は全国地方公共団体コード（`N03_007`）を使って `{コード}.geojson` と命名する。このコードは全国でユニークであるため、県をまたいでも衝突しない。

プリフェッチの仕組みとして、現在地の市町村が確定した時点で、その市町村に隣接する市町村のGeoJSONファイルをバックグラウンドで先読みする。これにより、市町村の境界を越えた瞬間にダウンロード待ちが発生しない。隣接関係は事前に計算した `adjacency.json`（市町村コード→隣接市町村コードの一覧）をアプリ起動時に一度だけ読み込んで利用する。

`adjacency.json` は事前処理スクリプト（Python + geopandas/shapely）をGoogle Cloud Shell上で実行して生成する。生成ロジックはN03の全市町村ポリゴンに対して `touches()` または `intersects()` で接触判定を行い、隣接する市町村コードの組み合わせをJSONとして書き出す。なお飛び地（例：北山村）など物理的に接触しない特殊ケースはPoCの段階では許容する。

市町村が前回の判定結果と異なった場合のみ「市町村切り替わり」と判断し、LLM呼び出しとUI更新を行う。

### 3.3 土地の解説生成（LLM）

市町村の切り替わりを検知したタイミングで、**Cloudflare Workers経由でClaude Haiku（claude-haiku-4-5-20251001）**にリクエストを送り、その土地の解説文を生成する。プロンプトには現在の都道府県名・市町村名・季節（春/夏/秋/冬）を渡す。季節は端末の日付から自動判定する（3〜5月：春、6〜8月：夏、9〜11月：秋、12〜2月：冬）。季節を渡す目的は、その土地の旬の食材・季節のイベントを解説に含めるためである。

生成する解説文の内容は、その土地の名物・旬の食材・季節のイベント・歴史・地形のいずれかに触れた、旅情を感じる短い文章（3〜4文程度）とする。LLM呼び出し中はスピナーを表示し、生成完了後にフェードインで解説文を表示する。

### 3.4 地図表示と通過軌跡

地図はLeaflet.jsを使って表示し、地理院タイル（淡色地図）をベースとする。現在地は地図上にマーカーで表示し、位置が更新されるたびにマーカーを移動させる。通過した軌跡はポリラインとして地図上に描画し、移動の記録を視覚的に確認できるようにする。地図の追従モードはON/OFFボタンで切り替えられるようにする。追従ONの場合は現在地を常に中心に表示し、追従OFFの場合はユーザーが自由にスクロール・ズーム操作できる。初期状態は追従ONとする。

### 3.5 制覇カウント

訪問済みの市町村数をカウントし、画面に表示する。同じ市町村を複数回訪れても制覇カウントは1としてカウントする。制覇データはlocalStorageに保存し、ブラウザを閉じても維持される。

### 3.6 画面レイアウト

画面の構成は上から順に以下の通りとする。まず**市町村名**を最も大きなフォントで表示する。次にLLMが生成した**土地の解説テキスト**を表示する（生成中はスピナー）。その下に**現在の移動速度**（km/h）を表示する。最下部に**地図**（現在地マーカー＋通過軌跡・追従ON/OFFボタン）と**制覇カウント**を表示する。

### 3.7 エラーハンドリング

エラーの種類に応じて以下の3方針で対処する。

**リトライで解決するエラー**（ネットワーク一時断・APIタイムアウト）については、自動リトライを行う。LLM呼び出しが失敗した場合はスピナーを表示したまま最大3回リトライし、それでも失敗した場合は最後に表示していた解説テキストをそのまま維持する。電波が途切れても画面が壊れないことを優先する。

**ユーザーに通知すべきエラー**（GPS権限拒否）については、「位置情報の許可をオンにしてください」というメッセージを画面に表示し、ユーザー自身に対処を促す。

**無視してよいエラー**（軌跡の座標記録失敗など）については、次の更新で自動的に回復するためリトライや通知は行わない。

### 3.8 パスワード認証

アプリ起動時にパスワード入力画面を表示する。入力されたパスワードはlocalStorageに保存し、次回以降は自動的に読み込む。Cloudflare WorkersはリクエストヘッダーにパスワードとしてWorker環境変数と照合し、一致しない場合は401エラーを返してAnthropicへのリクエストを転送しない。パスワード自体はCloudflare Workersの環境変数として管理し、コードには直書きしない。これにより、たまたまURLを踏んだ第三者がAPIを無断利用することを防ぐ。

## 4. 非機能要件

### 4.1 動作環境

対象ブラウザはiOS SafariおよびAndroid Chromeの最新版とする。スマホの縦画面（ポートレート）での利用を主に想定する。

### 4.2 パフォーマンス

市町村単位のGeoJSONファイルは1ファイルあたり数十KB〜数百KB程度となる見込みで、プリフェッチにより市町村切り替わり時のダウンロード待ちは発生しない。間引き処理の強度は、地図上の境界線精度と実測ファイルサイズのバランスを見ながら調整する。LLM呼び出しのレスポンスタイムはネットワーク状況に依存するが、スピナーを表示することでユーザーへの体感待ち時間を許容範囲にする。

### 4.3 セキュリティ

AnthropicのAPIキーおよびパスワードはいずれもCloudflare Workersの環境変数として管理し、ブラウザには一切露出しない。公開リポジトリにこれらの値をコミットしてはならない。CORSヘッダーでCloudflare PagesのドメインからのリクエストのみWorkerが受け付けるよう設定する。

### 4.4 ライセンス・利用規約

使用するすべての外部リソースのライセンスおよび利用規約への準拠を必須とする。地理院タイルは「地理院タイル」と出典を明示する。国土数値情報N03の利用時は「国土数値情報（行政区域データ）（国土交通省）を加工して作成」と表記する。

## 5. データ仕様

### 5.1 国土数値情報 N03（行政区域データ）

ダウンロード元はhttps://nlftp.mlit.go.jp/ksj/gml/datalist/KsjTmplt-N03-v3_1.html とする。ライセンスはPDL1.0で商用利用可。GeoJSONの各フィーチャーに含まれる `N03_001`（都道府県名）と `N03_004`（市区町村名）および `N03_007`（全国地方公共団体コード）を使用する。

### 5.2 隣接市町村マスタ（adjacency.json）

Google Cloud Shell上でPython（geopandas + shapely）を使い、N03の全市町村ポリゴン間の接触判定を行って生成する。データ構造は以下の通りとする。

```json
{
  "11234": ["11213", "08442", "11245"],
  "08442": ["11234", "08401", "08443"]
}
```

キーは市町村コード（`N03_007`）、値はそのコードに隣接する市町村コードの配列である。

### 5.3 地理院タイル

タイルURLは `https://cyberjapandata.gsi.go.jp/xyz/pale/{z}/{x}/{y}.png`（淡色地図）を使用する。ライセンスはPDL1.0で申請不要、出典明示のみでよい。参照元：https://maps.gsi.go.jp/development/ichiran.html

### 5.4 localStorageのデータ構造

```json
{
  "password": "xxxxxxxx",
  "visited": {
    "11234": { "name": "久喜市", "prefecture": "埼玉県", "firstVisit": "2026-04-22T10:00:00" }
  },
  "track": [
    { "lat": 35.681, "lon": 139.767, "ts": 1745000000000 }
  ]
}
```

## 6. 作業フロー

実装に入る前に以下の事前準備作業が必要である。

まず**Anthropic APIキーの発行**として、Anthropicコンソール（https://console.anthropic.com）でアカウントを作成し、クレジットをチャージしてAPIキーを発行する。次に**N03データの前処理**として、Google Cloud Shell上でN03全国データをダウンロードし、市町村単位のGeoJSONファイルへの分割・間引き処理・`adjacency.json` の生成を行い、Cloudflare Pagesに配置する。その後**Cloudflare Workersの構築**として、AnthropicへのAPIプロキシWorkerを作成してAPIキーとパスワードを環境変数として設定する。最後にフロントエンドのコーディングと実機確認を行う。

---

## 7. フェーズ計画

**フェーズ1（PoC・本ドキュメントの対象）**はGPS取得・市町村判定・LLM解説・地図軌跡・速度表示・制覇カウント・パスワード認証の実装を行う。

**フェーズ2**では道の駅の近隣表示（進行方向フィルタ ±60°、1200駅データ活用）を追加する。

**フェーズ3**ではGPXエクスポート・オフライン対応を追加する。

**フェーズ4**ではReact + Viteへの移行、または将来のReact Native移植を検討する。
